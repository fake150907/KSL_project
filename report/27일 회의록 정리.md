## 1. 회의 목적

본 회의는 현재 수어 인식 모델의 성능 문제를 점검하고, 향후 개발 방향을 결정하기 위해 진행하였다.
특히 기존 OpenPose 기반 keypoint 데이터와 실시간 처리에 적합한 MediaPipe 기반 landmark 데이터 간의 차이를 분석하고, 모델 재학습 및 검증 방향을 논의하였다.

---

## 2. 현재 프로젝트 진행 상황

현재 프로젝트는 8개 라벨을 기준으로 subset 데이터 구성을 완료한 상태이다.

대상 라벨은 다음과 같다.

```text
가다, 감사, 괜찮다, 배고프다, 병원, 아프다, 우유, 자다
```

현재까지 keypoint 좌표 기반으로 모델 학습을 수행하였으며, 여러 모델 구조를 실험한 결과 **CNN + GRU 구조가 가장 높은 성능**을 보였다.

GPU를 활용해 가능한 범위 내에서 모델 학습 및 성능 실험을 진행했으며, 일부 단어는 비교적 안정적으로 인식되는 결과를 보였다. 예를 들어 `가다`, `배고프다` 등은 상대적으로 잘 인식되었으나, 전체적으로는 오인식이 존재하였다.

---

## 3. 주요 문제점

## 3.1 인식 정확도 불안정

현재 모델은 일부 단어에서는 비교적 좋은 성능을 보이나, 전체 라벨에 대해 안정적인 인식 성능을 보이지는 못하고 있다.

특히 실제 웹캠 환경이나 유튜브 영상 클립을 입력했을 때, 학습 데이터에서 보였던 성능보다 낮은 결과가 나타났다.

---

## 3.2 프레임 처리 문제

현재 실시간 추론 과정에서는 긴 시간 동안 프레임을 저장하거나, 일정 구간을 기준으로 예측을 수행하는 방식이 사용되었다.

그러나 수어 동작은 짧은 시간 안에 발생하는 경우가 많기 때문에, 긴 구간을 그대로 분석하면 다음과 같은 문제가 발생한다.

* 동작 시작 구간과 종료 구간이 명확하지 않음
* 수어 동작 외의 불필요한 움직임이 포함됨
* 동일 동작을 반복하거나 멈춘 자세가 섞이면서 예측이 흔들림
* 마지막 프레임 또는 마지막 window 기준 예측이 실제 동작과 달라질 수 있음

따라서 향후에는 고정 길이 시퀀스를 사용하고, sliding window 방식과 동작 구간 분리 로직을 적용할 필요가 있다.

---

## 3.3 학습 데이터와 실제 입력 데이터의 불일치

현재 가장 큰 문제는 학습 데이터와 실제 추론 데이터의 좌표 체계가 다르다는 점이다.

기존 학습 데이터는 AIHub에서 제공한 **OpenPose 계열 keypoint**를 기반으로 구성되어 있다. 반면 웹캠이나 유튜브 영상 테스트에서는 **MediaPipe Holistic landmark**를 사용하고 있다.

즉, 구조는 다음과 같다.

```text
학습 데이터: OpenPose 기반 keypoint
실제 입력: MediaPipe 기반 landmark
```

이 경우 사람이 보기에는 같은 수어 동작이더라도, 모델 입장에서는 전혀 다른 좌표 분포로 인식될 수 있다.

추가로 MediaPipe에서는 z-depth 값이 포함되며, 좌표 스케일과 landmark 순서 역시 OpenPose와 다르기 때문에 단순히 입력 형태만 맞춘다고 해결되기 어렵다.

---

## 3.4 웹캠 테스트와 모델 성능 평가의 구분 필요

현재 일부 테스트는 웹캠 또는 유튜브 영상으로 진행되었다.
그러나 웹캠 테스트는 실제 사용 가능성을 확인하는 과정이지, 모델의 공식 성능 평가 기준으로 보기 어렵다.

모델 성능은 우선 다음 구조로 평가해야 한다.

```text
Train 데이터 → 모델 학습
Validation 데이터 → 모델 성능 평가
웹캠/유튜브 영상 → 실사용 가능성 테스트
```

따라서 AIHub Validation 데이터를 활용하여 동일한 데이터 분포 안에서 모델 정확도를 먼저 확인해야 한다.

---

## 4. 강사님 피드백 핵심 정리

강사님 피드백의 핵심은 다음과 같다.

첫째, 현재 프로젝트에서 가장 중요한 것은 **빠르게 베이스라인 모델을 구축하고 추론 가능성을 확인하는 것**이다. 전처리와 데이터 확장에만 집중하다 보면, 실제 모델 검증 없이 프로젝트가 종료될 위험이 있다. 

둘째, OpenPose와 MediaPipe는 각각 역할이 다르다. OpenPose는 기존 데이터셋과 구조가 맞기 때문에 정확도 기준 평가에 적합하지만, 실시간 처리에는 속도 문제가 있다. 반면 MediaPipe는 실시간 처리에는 적합하지만, 기존 OpenPose 기반 학습 데이터와 구조가 달라 성능 저하가 발생할 수 있다. 

셋째, OpenPose 기반 추론과 MediaPipe 기반 추론을 비교하여 최종 방향을 결정해야 한다. 즉, 단순히 한 가지 방식만 고집하기보다, 두 방식의 성능 차이를 확인한 뒤 프로젝트 목표에 맞는 방향을 선택해야 한다.

## 5. 모델 및 기술 방향 논의

## 5.1 현재 사용 모델

현재까지의 실험에서는 **CNN + GRU 기반 시퀀스 모델**이 가장 좋은 성능을 보였다.

따라서 당장 새로운 모델 구조를 무리하게 도입하기보다는, 기존 CNN + GRU 모델을 기준으로 데이터 입력 방식을 정리하고 성능을 검증하는 것이 우선이다.

---

## 5.2 향후 고려 가능한 모델

시간적 여유가 생긴다면 다음 모델 구조도 검토할 수 있다.

* Transformer 계열 모델
* Multi-head 구조
* CTC 기반 학습 방식
* 단어 인식과 문장 인식을 분리하는 구조

다만 현재 프로젝트는 MVP 단계이므로, 우선은 8개 단어 인식에 집중하는 것이 적절하다.

---

## 6. 향후 개발 방향

회의 결과, 향후 방향은 크게 두 가지로 정리되었다.

---

## 6.1 1안: MediaPipe 기준 재학습

### 개념

원본 동영상에서 MediaPipe landmark를 새로 추출하고, 해당 데이터를 기준으로 모델을 다시 학습하는 방식이다.

구조는 다음과 같다.

```text
원본 동영상
→ MediaPipe landmark 추출
→ numpy 좌표 데이터 생성
→ CNN + GRU 모델 재학습
→ 웹캠 MediaPipe 입력으로 추론
```

### 장점

* 학습 데이터와 실시간 입력 데이터의 좌표 체계를 통일할 수 있음
* 웹캠 기반 실시간 서비스에 적합함
* 근본적인 성능 개선 가능성이 큼

### 단점

* 원본 동영상이 필요함
* MediaPipe landmark 추출 및 데이터셋 재구성이 필요함
* 시간이 더 오래 걸림

### 판단

1안은 시간이 더 걸리지만, 실사용 환경까지 고려했을 때 가장 정석적인 접근이다.
최종적으로는 이 방향이 프로젝트의 핵심 해결책이 될 가능성이 높다.

---

## 6.2 2안: MediaPipe → OpenPose 호환 재매핑

### 개념

MediaPipe landmark를 기존 OpenPose 기반 모델이 기대하는 좌표 구조에 최대한 맞춰 변환한 뒤, 기존 모델 checkpoint를 그대로 활용하는 방식이다.

구조는 다음과 같다.

```text
웹캠/동영상
→ MediaPipe landmark 추출
→ OpenPose BODY_25 + hand 구조로 재매핑
→ 기존 CNN + GRU 모델로 추론
```

### 장점

* 기존 모델 checkpoint를 그대로 사용할 수 있음
* 원본 동영상이 없어도 빠르게 실험 가능함
* 특정 라벨 쏠림 현상이 좌표 순서 문제인지 확인하기 좋음

### 단점

* OpenPose와 MediaPipe는 완전히 같은 좌표 체계가 아니므로 완벽한 변환은 불가능함
* z-depth, confidence, landmark 의미 차이를 완전히 보정하기 어려움
* 일부 라벨에서는 여전히 오분류가 발생할 가능성이 큼

### 판단

2안은 빠른 실험용으로는 의미가 있지만, 최종 해결책으로 보기에는 한계가 있다.
따라서 2안은 “빠른 검증용”으로 활용하고, 최종 모델은 1안 MediaPipe 기준 재학습 방향으로 진행하는 것이 적절하다.

---

## 7. 실험 및 검증 계획

## Step 1. 기존 OpenPose 기반 모델 성능 평가

* 기존 keypoint subset 사용
* AIHub Validation 데이터로 정확도 측정
* CNN + GRU 모델 기준 성능 확인

목표는 현재 모델이 동일한 데이터 분포 안에서 어느 정도 성능을 내는지 확인하는 것이다.

---

## Step 2. MediaPipe → OpenPose 재매핑 실험

* MediaPipe landmark를 OpenPose 호환 구조로 재매핑
* 기존 모델 checkpoint로 추론
* 특정 라벨 쏠림 현상이 개선되는지 확인

이 실험은 빠르게 진행할 수 있으며, 좌표 순서 문제 여부를 파악하는 데 의미가 있다.

---

## Step 3. MediaPipe 기준 데이터셋 생성

* 8개 라벨에 해당하는 원본 동영상 확보
* MediaPipe Holistic 적용
* pose, left hand, right hand landmark 추출
* numpy 또는 npz 형식으로 저장
* 프레임 길이 통일

---

## Step 4. MediaPipe 기준 모델 재학습

* 기존 CNN + GRU 구조 유지 ==> 구조 유지를 하되 추가적인 모델 분석도 진행해서 비교진행
* 입력 데이터만 MediaPipe 기반으로 변경
* Train/Validation 분리 후 성능 평가
* confusion matrix를 통해 라벨별 오분류 확인

---

## Step 5. 웹캠 실사용 테스트

* Validation 성능 확인 후 웹캠 테스트 진행
* 실제 환경에서의 지연 시간, 오인식, 노이즈 영향을 확인
* confidence threshold 및 sliding window 방식 적용

---

## 8. 추가 보완 사항

## 8.1 프레임 처리 개선

현재처럼 긴 시간 단위로 프레임을 저장하고 분석하는 방식은 적절하지 않다.
향후에는 다음 방식을 적용한다.

* 고정 길이 시퀀스 사용

  * 예: 32프레임 또는 64프레임
* sliding window 적용
* 동작 시작/종료 구간 분리
* 마지막 window만 보지 않고 average 또는 majority 결과 함께 활용

---

## 8.2 좌표 정규화

MediaPipe의 x, y landmark는 기본적으로 이미지 크기 기준 0~1 범위로 제공된다.
그러나 실시간 웹캠 환경에서는 사용자 위치, 카메라 거리, 신체 크기, 화면 내 이동에 따라
동일한 행동이라도 좌표 분포가 달라질 수 있으므로 추가적인 정규화가 필요하다.

적용 가능한 방식은 다음과 같다.

* 손목, 어깨 중심, 골반 중심 등을 기준으로 한 상대좌표 변환
* 손 크기 또는 어깨 너비 기준의 스케일 정규화
* 프레임 간 이동량(dx, dy, dz) feature 추가
* 관절 간 거리, 각도, 손가락 굽힘 feature 활용
* z-depth는 상대 깊이로 사용하되, 절대 거리값으로 해석하지 않도록 주의

---

## 8.3 노이즈 제거

웹캠 환경에서는 사용자의 불필요한 움직임이 포함될 수 있다.
따라서 다음 로직을 추가할 필요가 있다.

* confidence threshold 적용
* 일정 프레임 이상 같은 라벨이 유지될 때만 출력
* 동일 라벨 반복 출력 방지
* 예측 결과 smoothing 적용

---

## 8.4 데이터 품질 통일

MediaPipe 기준 재학습을 위해 새롭게 동영상을 확보할 경우, 데이터 품질을 통일해야 한다.

권장 기준은 다음과 같다.

* 정면 촬영 우선
* 동일한 거리 유지
* 밝은 조명 환경
* 시작/끝 동작을 명확히 촬영
* 한 영상에는 하나의 수어 동작만 포함

---

## 9. 최종 결론

현재 프로젝트는 8개 라벨 기준 subset 데이터와 CNN + GRU 모델을 통해 기본적인 수어 인식 가능성을 확인하였다.
그러나 실제 웹캠 또는 유튜브 영상 환경에서는 학습 데이터와 입력 데이터의 좌표 체계 차이, 프레임 처리 문제, 노이즈 문제로 인해 인식 성능이 불안정하게 나타났다.

따라서 향후 방향은 다음과 같이 정리한다.

```text
OpenPose 기반 모델:
기준 성능 확인 및 비교용으로 활용

MediaPipe → OpenPose 재매핑:
빠른 실험 및 좌표 순서 문제 확인용으로 활용

MediaPipe 기준 재학습:
실시간 웹캠 서비스 구현을 위한 최종 핵심 방향
```

즉, 2안은 빠르게 확인할 수 있는 실험으로 진행하되, 최종적으로는 **1안 MediaPipe 기준 재학습**을 중심으로 개발을 이어가는 것이 적절하다.

# 회의 핵심 요약

**완벽한 데이터셋 확장보다 빠른 모델 검증이 우선이다.**
현재 성능 문제는 모델 구조만의 문제가 아니라, **프레임 처리 방식과 OpenPose/MediaPipe 입력 구조 차이**에서 발생하고 있다.
따라서 기존 OpenPose 기반 모델은 기준 성능 확인용으로 유지하고, 실시간 웹캠 서비스 구현을 위해서는 **MediaPipe 기준 재학습**으로 전환하는 것이 가장 현실적인 방향이다.
