# AI Hub 수어영상 데이터 어노테이션 구조 정리

이 문서는 AI Hub 수어영상 데이터셋에서 제공하는 `morpheme JSON`, `keypoint JSON`, 원천 동영상의 관계와 각 필드의 의미를 정리한 문서다.

참고 자료:

- `작업지시서/02. [자연어영역] 수어 영상.pdf`
- `작업지시서/004.수어영상_데이터_구축_가이드라인.pdf`
- 실제 다운로드한 AI Hub `WORD` 계열 morpheme/keypoint JSON 파일

## 전체 구조

AI Hub 수어영상 데이터는 크게 세 종류로 나뉜다.

| 데이터 | 예시 파일 | 역할 |
|---|---|---|
| 원천 동영상 | `NIA_SL_WORDxxxx_REAL01_D.mp4` | 사람이 실제 수어를 수행한 영상 |
| morpheme JSON | `NIA_SL_WORDxxxx_REAL01_D_morpheme.json` | 정답 라벨, 형태소 구간, 부가 어노테이션 |
| keypoint JSON | `NIA_SL_WORDxxxx_REAL01_D_000000000000_keypoints.json` | 프레임별 몸/손/얼굴 랜드마크 좌표 |

현재 MVP에서는 원천 동영상 자체는 사용하지 않고, 다음처럼 학습 데이터를 만든다.

```text
입력 X = keypoint JSON의 좌표 시퀀스
정답 y = morpheme JSON의 attributes.name
```

즉 모델은 동영상 파일을 직접 보는 것이 아니라, 동영상에서 미리 추출된 랜드마크 좌표를 보고 `감사`, `병원`, `아프다` 같은 수어 표현 클래스를 분류한다.

## 수지와 비수지

수어는 손동작만으로 구성되지 않는다. 손으로 표현되는 요소와 손이 아닌 요소를 구분해서 볼 수 있다.

| 용어 | 의미 | 예시 | 현재 MVP 반영 여부 |
|---|---|---|---|
| 수지 | 손으로 표현되는 수어 요소 | 손 모양, 손 위치, 손가락 움직임, 양손 관계 | 반영 |
| 비수지 | 손이 아닌 신체/표정 요소 | 얼굴 표정, 눈썹, 시선, 고개 움직임, 상체 방향, 입모양 | 초기 MVP에서는 제외 |

`수지`는 AI Hub keypoint JSON에서 주로 손 랜드마크 좌표로 표현된다.

```text
hand_left_keypoints_2d
hand_right_keypoints_2d
```

`비수지`는 morpheme JSON의 `attributes.attribute`에 어노테이션될 수 있고, keypoint JSON에서는 얼굴/상체 관련 좌표와 연결될 수 있다.

```text
attributes.attribute
face_keypoints_2d
pose_keypoints_2d 일부
```

다만 현재 MVP 요구사항은 얼굴 표정/상체/비수지 요소 고도화를 제외하므로, `attributes.attribute`와 `face_keypoints_2d`는 학습 입력에서 제외한다.

주의할 점은 `attributes.attribute` 필드 자체가 항상 비수지를 뜻하지는 않는다는 것이다. PDF 표에서는 `attributes(attributes)`를 비수지 정보로 설명하지만, 실제 예시에는 `"두손 검지 모아서"`처럼 손 모양이나 손동작을 설명하는 값도 들어 있다. 따라서 개발 관점에서는 다음처럼 이해하는 것이 안전하다.

```text
attributes.name = 정답 형태소/라벨
attributes.attribute = 해당 라벨에 대한 부가 어노테이션
```

그리고 `attributes.attribute`의 내용은 경우에 따라 나뉜다.

| attribute 내용 | 해석 |
|---|---|
| `두손 검지 모아서` | 손 모양/손동작 설명이므로 수지 정보에 가까움 |
| `눈썹 올림`, `고개 끄덕임`, `시선 이동` | 손이 아닌 표현이므로 비수지 정보 |
| 기타 설명 | 라벨을 보조하는 사람이 읽는 어노테이션으로 취급 |

현재 모델은 이 문장형 어노테이션을 직접 학습하지 않는다. 실제 수지 정보는 `hand_left_keypoints_2d`, `hand_right_keypoints_2d`의 좌표 패턴으로 학습한다.

## Morpheme JSON

morpheme JSON은 수어 표현의 정답 라벨과 발화 구간 정보를 담는다.

실제 예시:

```json
{
  "metaData": {
    "url": "https://.../NIA_SL_WORD0001_REAL01_D.mp4",
    "name": "NIA_SL_WORD0001_REAL01_D.mp4",
    "duration": 4.867,
    "exportedOn": "2020/12/10"
  },
  "data": [
    {
      "start": 1.743,
      "end": 3.103,
      "attributes": [
        {
          "name": "고민"
        }
      ]
    }
  ]
}
```

필드 의미:

| 필드 | 의미 | 현재 프로젝트 사용 여부 |
|---|---|---|
| `metaData.url` | 원천 수어 동영상 경로 | 참고용 |
| `metaData.name` | 원천 수어 동영상 파일명 | sample id 매칭에 참고 |
| `metaData.duration` | 동영상 전체 길이 | `start/end`를 프레임 인덱스로 바꿀 때 사용 |
| `exportedOn` | JSON 다운로드/생성 날짜 | 미사용 |
| `data[].start` | 해당 형태소/단어 발화 시작 시각 | 사용 |
| `data[].end` | 해당 형태소/단어 발화 종료 시각 | 사용 |
| `attributes[].name` | 형태소 정보, 즉 정답 라벨 | 사용 |
| `attributes[].attribute` | 부가 어노테이션. 내용에 따라 수지 설명 또는 비수지 설명일 수 있음 | MVP 범위에서는 제외 |

`attributes.attribute` 예시:

```json
{
  "attributes": [
    {
      "name": "만나다",
      "attribute": [
        "두손 검지 모아서"
      ]
    }
  ]
}
```

여기서 `name`은 정답 형태소/표현이고, `attribute`는 그 표현을 설명하는 부가 어노테이션이다.

이 예시의 `"두손 검지 모아서"`는 얼굴 표정이나 시선 같은 비수지가 아니라, 두 손의 검지를 모으는 손동작 설명이다. 따라서 이 값은 비수지라기보다 수지적 설명으로 보는 것이 더 자연스럽다.

다만 모델 학습에서는 `"두손 검지 모아서"`라는 문장을 입력으로 쓰지 않는다. 실제 손동작은 keypoint JSON의 손 좌표에서 가져온다.

```text
문장형 설명: attributes.attribute = "두손 검지 모아서"
실제 모델 입력: hand_left_keypoints_2d + hand_right_keypoints_2d 좌표
```

현재 프로젝트는 표현 분류 MVP이므로 `attributes.name`만 라벨로 사용하고, `attributes.attribute`는 분석 참고용으로만 둔다.

중요한 점:

```text
duration = 동영상 전체 길이
start/end = 동영상 안에서 실제 수어 형태소가 발화되는 구간
```

예를 들어:

```text
label: 우유
start: 1.708
end: 3.488
duration: 4.367
```

이 뜻은 전체 동영상은 4.367초이고, 실제 `우유` 수어 동작 구간은 1.708초부터 3.488초까지라는 의미다.

## Keypoint JSON

keypoint JSON은 원천 동영상의 각 프레임에서 추출된 사람의 랜드마크 좌표를 담는다.

실제 예시의 핵심 구조:

```json
{
  "version": 1.3,
  "people": {
    "face_keypoints_2d": [x, y, confidence, ...],
    "pose_keypoints_2d": [x, y, confidence, ...],
    "hand_left_keypoints_2d": [x, y, confidence, ...],
    "hand_right_keypoints_2d": [x, y, confidence, ...],
    "face_keypoints_3d": [x, y, z, confidence, ...],
    "pose_keypoints_3d": [x, y, z, confidence, ...],
    "hand_left_keypoints_3d": [x, y, z, confidence, ...],
    "hand_right_keypoints_3d": [x, y, z, confidence, ...]
  }
}
```

PDF 설명 기준 필드 의미:

| 필드 | 의미 | 현재 프로젝트 사용 여부 |
|---|---|---|
| `hand_left_keypoints_2d` | 왼손 특징점 좌표 값 | 사용 |
| `hand_right_keypoints_2d` | 오른손 특징점 좌표 값 | 사용 |
| `pose_keypoints_2d` | 몸통, 팔 등 자세 특징점 좌표 값 | 사용 |
| `face_keypoints_2d` | 얼굴 특징점 좌표 값 | MVP 범위에서는 제외 |
| `*_keypoints_3d` | 3차원 특징점 좌표 | 현재는 2D 중심이라 제외 |

좌표 배열은 보통 다음처럼 3개 단위로 반복된다.

```text
x, y, confidence, x, y, confidence, ...
```

수지에 해당하는 손 좌표 예시:

```json
{
  "version": 1.3,
  "people": {
    "hand_left_keypoints_2d": [
      1134.48, 962.59, 1,
      1108.83, 978.24, 1,
      1091.90, 1007.28, 1
    ],
    "hand_right_keypoints_2d": [
      646.34, 963.82, 1,
      671.91, 982.70, 1,
      686.84, 1015.65, 1
    ]
  }
}
```

위 배열에서 3개 값이 하나의 랜드마크 점이다.

```text
1134.48, 962.59, 1
= x좌표, y좌표, confidence
```

따라서 손 keypoint 배열은 다음과 같은 손 관절점들의 연속이다.

```text
손목 좌표
엄지 관절 좌표들
검지 관절 좌표들
중지 관절 좌표들
약지 관절 좌표들
소지 관절 좌표들
```

현재 MVP는 얼굴 표정/비수지 고도화를 제외하므로, 학습 입력에는 다음만 사용한다.

```text
pose_keypoints_2d
hand_left_keypoints_2d
hand_right_keypoints_2d
```

이 중에서 손동작, 즉 수지에 가장 직접적으로 해당하는 것은 다음 두 필드다.

```text
hand_left_keypoints_2d
hand_right_keypoints_2d
```

`pose_keypoints_2d`는 손목, 팔꿈치, 어깨 등 팔/상체 위치를 보조적으로 제공한다. 손만으로 구분하기 어려운 동작 방향이나 팔의 이동을 잡는 데 도움이 된다.

## 파일 매칭 방식

morpheme JSON과 keypoint JSON은 파일명 기반 sample id로 매칭한다.

예:

```text
morpheme:
NIA_SL_WORD0815_REAL01_D_morpheme.json

keypoint folder:
NIA_SL_WORD0815_REAL01_D/

keypoint frames:
NIA_SL_WORD0815_REAL01_D_000000000000_keypoints.json
NIA_SL_WORD0815_REAL01_D_000000000001_keypoints.json
...
```

매칭에 사용하는 sample id:

```text
NIA_SL_WORD0815_REAL01_D
```

즉:

```text
NIA_SL_WORD0815_REAL01_D_morpheme.json
        ↓
NIA_SL_WORD0815_REAL01_D
        ↓
NIA_SL_WORD0815_REAL01_D/*.json
```

## 파일명 규칙과 공통 Prefix

AI Hub 수어영상 데이터는 원천 영상, morpheme JSON, keypoint JSON이 같은 파일명 prefix를 공유한다. 이 prefix가 정답 라벨과 좌표 시퀀스를 연결하는 핵심 키다.

예:

```text
원천 영상:
NIA_SL_WORD0001_REAL01_F.mp4

morpheme:
NIA_SL_WORD0001_REAL01_F_morpheme.json

keypoint:
NIA_SL_WORD0001_REAL01_F_000000000000_keypoints.json
NIA_SL_WORD0001_REAL01_F_000000000001_keypoints.json
...
```

세 파일에서 공통으로 반복되는 부분은 다음이다.

```text
NIA_SL_WORD0001_REAL01_F
```

이 공통 prefix가 같으면 같은 샘플로 본다.

| 파일명 부분 | 의미 |
|---|---|
| `NIA_SL` | AI Hub 수어 데이터 식별 prefix |
| `WORD0001` | 수어 단어 순번 |
| `REAL01` | 촬영 방식 및 수어 제공자 번호 |
| `F` | 촬영각도 표식자 |
| `_morpheme` | 형태소/부가 어노테이션 파일 |
| `000000000000` | keypoint 프레임 번호 |
| `_keypoints` | 프레임별 keypoint 좌표 파일 |

즉 영상, morpheme, keypoint는 다음 기준으로 연결된다.

```text
WORD 번호 + 촬영방식/제공자 번호 + 촬영각도
```

예를 들어:

```text
NIA_SL_WORD0001_REAL01_F.mp4
NIA_SL_WORD0001_REAL01_F_morpheme.json
NIA_SL_WORD0001_REAL01_F_000000000000_keypoints.json
```

은 모두 같은 샘플 `NIA_SL_WORD0001_REAL01_F`에 속한다.

keypoint 파일만 뒤에 프레임 번호가 추가된다.

```text
NIA_SL_WORD0001_REAL01_F_000000000000_keypoints.json
                          └──────────── 첫 번째 프레임 번호
```

현재 프로젝트의 매칭 방식도 이 규칙을 따른다.

```text
1. morpheme 파일명에서 `_morpheme` 제거
2. sample id 생성: NIA_SL_WORD0815_REAL01_D
3. 같은 이름의 keypoint 폴더 찾기
4. 폴더 안의 `*_keypoints.json` 프레임 파일들을 시간순으로 읽기
5. morpheme의 `attributes.name`을 정답 라벨로 붙이기
```

이 규칙이 중요한 이유는 morpheme JSON은 정답 라벨을 가지고 있고, keypoint JSON은 좌표만 가지고 있기 때문이다. 두 데이터를 같은 샘플로 묶으려면 파일명 prefix가 반드시 필요하다.

## 촬영각도 표식자

파일명 끝의 `F/U/D/R/L`은 JSON 내부 필드가 아니라 파일명에 포함된 촬영각도 표식자다.

예:

```text
NIA_SL_WORD0815_REAL01_D
NIA_SL_WORD0815_REAL01_F
NIA_SL_WORD0815_REAL01_L
NIA_SL_WORD0815_REAL01_R
NIA_SL_WORD0815_REAL01_U
```

위 5개는 모두 같은 단어 `우유`의 샘플이지만, 서로 다른 촬영각도에서 촬영된 데이터다.

| 표식 | 의미 |
|---|---|
| `F` | Front, 정면 |
| `U` | Up, 위쪽 각도 |
| `D` | Down, 아래쪽 각도 |
| `R` | Right, 오른쪽 각도 |
| `L` | Left, 왼쪽 각도 |

촬영각도는 정답 라벨은 아니지만 모델 성능에 영향을 줄 수 있다. 같은 수어 표현이라도 카메라 각도가 달라지면 keypoint 좌표 분포가 달라지기 때문이다.

예를 들어 같은 `우유` 동작이라도:


```text
F: 정면에서 본 손/팔 위치
R/L: 옆쪽에서 본 손/팔 위치
U/D: 위/아래 각도에서 본 y좌표와 상체 비율
```

처럼 좌표 패턴이 달라질 수 있다.

현재 프로젝트에서는 촬영각도를 모델 입력 feature로 직접 넣지는 않는다. 대신 manifest와 성능 리포트에 `angle` 컬럼을 명시해 데이터 분석과 검증 분할에 사용한다.

현재 manifest 예:

```text
sample_id,label,angle,start,end,duration,split,keypoint_path
NIA_SL_WORD0815_REAL01_D,우유,D,1.708,3.488,4.367,train,...
NIA_SL_WORD0815_REAL01_U,우유,U,1.708,3.488,4.367,validation,...
```

현재 split 전략은 간단한 경량 검증을 위해 `U` 각도를 validation으로 둔다. 따라서 현재 baseline 검증 정확도는 주로 “학습에 사용하지 않은 위쪽 각도 샘플을 얼마나 맞히는가”를 보는 지표에 가깝다.

현재 각도별 검증 결과 예:

```text
angle: U
validation samples: 12
accuracy: 0.8333
```

웹캠 실시간 데모는 보통 정면 카메라 입력이므로, 추후에는 `F` 정면 샘플 중심 실험과 전체 각도 혼합 실험을 분리해서 비교하는 것이 좋다.

## 현재 전처리 방식

PDF 설명 자료를 반영하여 현재 전처리는 다음 순서로 수행한다.

1. morpheme JSON에서 `attributes.name`을 읽어 정답 라벨로 사용한다.
2. morpheme JSON에서 `start`, `end`, `duration`을 읽는다.
3. 파일명에서 촬영각도 `F/U/D/R/L`을 추출해 manifest의 `angle` 컬럼에 저장한다.
4. 같은 sample id를 가진 keypoint 프레임 폴더를 찾는다.
5. 전체 keypoint 프레임 중 `start/end` 구간에 해당하는 프레임만 선택한다.
6. 선택된 프레임에서 `pose`, `left hand`, `right hand` 2D 좌표를 읽는다.
7. 좌표를 정규화한다.
8. 시퀀스 길이를 32프레임으로 통일한다.
9. 모델 입력 tensor로 저장한다.

현재 tensor 형태:

```text
(samples, sequence_length, feature_count)
= (60, 32, 201)
```

의미:

| 차원 | 의미 |
|---|---|
| `60` | 추출된 실제 keypoint 샘플 수 |
| `32` | 샘플당 고정 프레임 수 |
| `201` | pose + left hand + right hand 좌표 feature 수 |

선택 기준:

- 생활형 데모에 적합한 표현
- keypoint zip `01_real_word_keypoint.zip` 안에서 실제 좌표 샘플과 매칭 가능

## 분석 과정

구조는 다음 순서로 확인했다.

1. AI Hub 파일 목록 확인
   - `real_word_video.zip`
   - `real_word_morpheme.zip`
   - `real_word_keypoint.zip`

2. morpheme JSON 실제 파일 열람
   - `metaData.name`, `metaData.url`에 동영상 정보 존재 확인
   - `data[].attributes[].name`에 정답 라벨 존재 확인
   - `start/end`에 발화 구간 존재 확인

3. keypoint JSON 실제 파일 열람
   - `people.pose_keypoints_2d`
   - `people.hand_left_keypoints_2d`
   - `people.hand_right_keypoints_2d`
   - `people.face_keypoints_2d`
   - 각 배열이 랜드마크 좌표 배열임을 확인

4. AI Hub 제공 PDF와 대조
   - PDF의 `json 포맷 요소` 표에서 필드 의미 확인
   - `start/end`가 형태소 시작/종료 시각임을 확인
   - `attributes.name`이 형태소 정보임을 확인
   - `hand_left/right`, `face`, `pose` keypoint 의미 확인

5. 전처리 코드 반영
   - 전체 프레임이 아니라 `start/end` 구간의 keypoint만 사용하도록 수정
   - 파일명 끝의 `F/U/D/R/L` 촬영각도를 manifest의 `angle` 컬럼으로 저장
   - baseline 리포트에 각도별 validation accuracy를 기록

## 현재 성능

실제 keypoint subset으로 baseline 모델을 학습한 결과:


주의:
