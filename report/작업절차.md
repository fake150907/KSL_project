# 작업절차

## 목적

이 문서는 이번 프로젝트에서 실제로 진행한 작업 흐름을 한 번에 다시 따라갈 수 있도록 정리한 절차 메모다.

범위는 다음을 포함한다.

- AIHub WORD morpheme 다운로드 및 확인
- morpheme 기반 라벨 분석
- 소규모 생활형 subset 선정
- keypoint subset 추출 및 병합 흐름
- 전처리, 학습, 추론
- Streamlit UI 실행과 디버깅

## 1. 기본 방향

초기 목표는 AIHub 수어영상 전체를 한 번에 처리하는 것이 아니라, `WORD` 계열만으로 경량 MVP를 먼저 만드는 것이었다.

초기 제외 범위:

- `SEN`
- 원천 비디오 직접 학습
- `CROWD`
- `SYN`
- 고도화된 비수지 모델링

초기 포함 범위:

- `WORD morpheme JSON`
- `WORD keypoint JSON`
- 소수 클래스 기반 end-to-end 검증

## 2. AIHub 데이터 다운로드 전략

AIHub 데이터셋은 먼저 morpheme만 받아 라벨 분포를 확인하고, 그 다음 필요한 keypoint만 선별적으로 받는 방향으로 진행했다.

참고:

- [README.md](C:/github/ai-project-01/README.md)
- [download_aihub_dataset.md](C:/github/ai-project-01/scripts/download_aihub_dataset.md)
- [aihub_filekeys_103.md](C:/github/ai-project-01/scripts/aihub_filekeys_103.md)

기본 디렉토리 구조:

```text
data/raw/
  Training/REAL/WORD/real_word_keypoint/
  Training/REAL/WORD/real_word_morpheme/
  Validation/REAL/WORD/real_word_keypoint/
  Validation/REAL/WORD/real_word_morpheme/
```

API 키는 소스에 넣지 않고 환경변수로 사용한다.

```powershell
$env:AIHUB_APIKEY="YOUR_KEY"
```

먼저 morpheme만 받는 권장 명령:

```powershell
.\scripts\download_aihub_word_morpheme.ps1
```

그 이유:

- morpheme는 용량이 작다
- 라벨 후보 확인이 먼저 가능하다
- keypoint는 대용량이라 subset 전략이 정해진 뒤 받는 편이 안전하다

## 3. morpheme 분석

라벨 추출은 [extract_labels.py](C:/github/ai-project-01/src/data/extract_labels.py) 로 수행한다.

대표 명령:

```powershell
python -m src.data.extract_labels --quick_test --max_classes 8
```

주요 출력물:

- [label_candidates.csv](C:/github/ai-project-01/data/label_candidates.csv)
- [selected_labels_small.json](C:/github/ai-project-01/data/selected_labels_small.json)

현재 로컬 데이터 기준 확인 결과:

- `data/raw` 아래 morpheme JSON 수: `9079`
- 현재 로컬 `Validation morpheme JSON`: `0`
- 고유 라벨 수: `1723`
- 중요한 전제: 현재 로컬에 있는 morpheme은 사실상 `REAL01` 기준만 있음
- 따라서 현재 생성되는 `selected_label_targets.csv` 도 `REAL01` sample id 기준만 포함함
- 즉 `02_real_word_keypoint.zip`, `03_real_word_keypoint.zip` 같은 `REAL02`, `REAL03` 계열 keypoint는 현재 morpheme만으로는 직접 매칭되지 않음

즉 현재 워크스페이스 기준으로는:

- morpheme 샘플은 `9079개`
- 실제 고유 단어/표현 라벨은 `1723개`

여기서 중요한 구분:

- `WORDxxxx` 번호 체계와 샘플 수는 다르다
- 샘플 수와 고유 라벨 수는 다르다
- 같은 라벨이 여러 샘플, 여러 각도로 반복 등장할 수 있다

## 5. morpheme 기반 target 추출

선정된 8개 클래스에 대해 sample id 목록을 먼저 만든다.

명령:

```powershell
python scripts/export_selected_targets.py
```

출력:

- [selected_label_targets.csv](C:/github/ai-project-01/data/selected_label_targets.csv)

현재 기준 target 수:

```text
가다      20
감사       5
괜찮다      5
배고프다    10
병원       5
아프다      5
우유      10
자다      15
총 75
```

이 파일은 작업자들이 어떤 sample id를 찾아야 하는지 알려주는 기준표 역할을 한다.

중요:

- 현재 로컬 기준 `selected_label_targets.csv` 는 `REAL01` 기준 target 목록이다
- 따라서 `01_real_word_keypoint.zip` 과는 바로 연결되지만, `02_real_word_keypoint.zip` 같은 다른 `REALxx` zip과는 현재 상태에서 바로 연결되지 않는다
- 이 점을 잊고 `02`, `03` zip을 같은 방식으로 바로 병합하려고 하면 `0개 추출`이 나올 수 있다

## 12. Streamlit 디버깅에서 확인한 점

어제 진행한 주요 확인 내용:

- 휴대폰 카메라 입력은 프레임이 자주 끊겼다
- 그래서 실시간 인식용 윈도우가 충분히 쌓이지 않는 문제가 있었다
- 카메라 오버레이에는 예측 라벨이 뜨는 경우가 있었음
- 하지만 `수어 -> 텍스트` 박스 자동 반영은 끝내 안정적으로 해결되지 않았음

실험한 조정:

- 카메라 해상도/프레임레이트 축소
- `frame_stride` 추가
- `window_size` 축소
- `stable_min_count` 조정
- `draw_pose` 끄기
- `sign_text` 자동 commit/fragment/rerun 반영 시도

현재 상태 요약:

- 실시간 오버레이 예측은 일부 동작
- 자동 텍스트 반영은 여전히 불안정
- 이 문제는 별도 작업지시서로 Claude Code에게 넘길 수 있도록 정리함

- [ClaudeCode_app.py_자동입력_수정_작업지시서.md](</C:/github/ai-project-01/작업지시서/ClaudeCode_app.py_자동입력_수정_작업지시서.md>)
