# 현재 모델이 보는 정보와 한계

## 핵심 요약

현재 모델은 원본 RGB 영상을 직접 보는 것이 아니라, MediaPipe가 추출한 상체/양손 좌표 시퀀스를 보고 수어 단어를 예측한다.

```text
입력 형태: 32프레임 x 225개 feature

225개 feature =
(pose 33 + left_hand 21 + right_hand 21) landmark x (x, y, z)
= 75 landmark x 3
```

즉 손가락 좌표는 이미 포함되어 있다. 다만 손가락 각도, 손바닥 방향, 손끝 거리, 속도 같은 의미 feature를 별도로 만들어 넣은 상태는 아니다.

## 현재 작업 기준

모델 개선 담당자는 아래 기준을 먼저 고정해서 이해해야 한다.

| 항목 | 현재 기준 |
|---|---|
| 대상 단어 | `가다`, `감사`, `괜찮다`, `배고프다`, `병원`, `아프다`, `우유`, `자다` |
| 라벨 수 | 8개 |
| 학습 데이터셋 | `FILE01_16_FULL` 통합 데이터셋 |
| 학습 샘플 수 | 484개 |
| 외부 validation 데이터셋 | `VALIDATION_MVP_FULL` |
| 외부 validation 샘플 수 | 150개 |
| 입력 shape | `(samples, 32, 225)` |
| sequence length | 32프레임 |
| feature count | 225 |
| feature layout | `mediapipe_xyz` |
| 좌표 구성 | `pose33 + left_hand21 + right_hand21`, 각 landmark `[x, y, z]` |
| 정규화 | 어깨 중심 기준, 어깨 너비 scale |
| 추천 모델 | CNN-GRU |
| 비교 모델 | LSTM |

라벨 index 순서는 반드시 유지해야 한다.

| index | label |
|---:|---|
| 0 | 가다 |
| 1 | 감사 |
| 2 | 괜찮다 |
| 3 | 배고프다 |
| 4 | 병원 |
| 5 | 아프다 |
| 6 | 우유 |
| 7 | 자다 |

라벨 순서가 바뀌면 모델 출력 index가 다른 단어로 해석된다. 학습, 평가, 웹 적용에서 같은 label map을 써야 한다.

## 데이터 구성

학습 데이터셋 `FILE01_16_FULL`은 FILE 단위로 모은 MVP mp4를 MediaPipe 전처리한 결과다.

```text
샘플 수: 484
X.shape: (484, 32, 225)
라벨 수: 8
```

라벨별 학습 샘플 수는 균등하지 않다.

| label | samples |
|---|---:|
| 가다 | 128 |
| 감사 | 32 |
| 괜찮다 | 32 |
| 배고프다 | 64 |
| 병원 | 32 |
| 아프다 | 32 |
| 우유 | 64 |
| 자다 | 100 |

외부 validation 데이터셋은 학습에 사용하지 않은 사람/샘플로 모델 일반화 성능을 보기 위한 holdout이다.

```text
VALIDATION_MVP_FULL
샘플 수: 150
X.shape: (150, 32, 225)
실패 샘플: 0
주요 REAL: REAL17, REAL18
```

따라서 내부 validation 100%보다 외부 validation 79.33%가 실제 서비스/새 사람 대응력에 더 가까운 지표다.

## 전처리에서 실제로 한 일

전처리는 원본 mp4 전체를 그대로 쓰지 않고, manifest에 있는 수어 의미 구간을 기준으로 자른 뒤 MediaPipe 좌표를 추출한다.

| 단계 | 현재 처리 |
|---|---|
| 입력 | mp4 영상 + manifest의 `start`, `end`, `duration`, `label`, `sample_id` |
| 구간 선택 | `start/end`를 frame index로 변환해 수어 의미 구간만 읽음 |
| 프레임 읽기 | `frame_step=1`, 즉 선택 구간의 모든 프레임을 처리 |
| 좌표 추출 | MediaPipe Holistic 결과에서 pose/left hand/right hand 사용 |
| face 사용 | Holistic을 돌리지만 face landmark는 모델 입력에 넣지 않음 |
| 검출 기준 | `min_detection_confidence=0.3`, `min_tracking_confidence=0.3` |
| frame feature | `pose33 + left_hand21 + right_hand21`의 `[x, y, z]` |
| 누락 landmark | 해당 slot을 0으로 채움 |
| 정규화 | 어깨 중심 기준으로 좌표 중심 이동, 어깨 너비로 scale |
| 길이 맞춤 | 32프레임보다 길면 균등 샘플링, 짧으면 뒤쪽을 0 padding |
| 저장 | `np.savez_compressed`로 `X`, `y`, `labels`, `sample_ids` 등 저장 |

정규화는 사람의 화면 위치와 체형 차이를 줄이기 위한 핵심 처리다.

```text
center = 양 어깨 midpoint
scale = 양 어깨 거리

어깨가 잡히지 않으면:
1. 골반 midpoint/골반 너비
2. valid landmark 평균/bbox 대각선
순서로 fallback
```

z 좌표도 같은 center/scale로 변환한다. 다만 MediaPipe z는 절대 깊이값이 아니라 상대 깊이 성격이 강하므로, 거리 단위로 해석하면 안 된다.

## NPZ 안에 들어 있는 의미

모델 개선자는 NPZ를 단순 배열로만 보면 안 된다. 각 축은 아래 의미를 가진다.

```text
X[i] = i번째 영상 샘플
X[i, t] = i번째 영상의 t번째 시간 step
X[i, t, :] = 그 시간 step의 pose/양손 좌표 feature 225개
```

225개 feature의 순서는 아래 기준이다.

```text
pose 33개 landmark x,y,z
left_hand 21개 landmark x,y,z
right_hand 21개 landmark x,y,z
```

즉 한 프레임 안에서 먼저 몸 pose가 나오고, 그 다음 왼손, 오른손이 이어진다. 이 순서가 바뀌면 기존 모델과 호환되지 않는다.

## 학습에서 실제로 한 일

현재 추천 모델은 CNN-GRU다. 같은 NPZ로 LSTM도 비교 학습했다.

| 항목 | CNN-GRU 현재 설정 |
|---|---|
| model_type | `cnn_gru` |
| 입력 크기 | 225 |
| 출력 클래스 | 8 |
| epochs | 50 |
| batch_size | 16 |
| learning_rate | 0.001 |
| optimizer | Adam |
| loss | CrossEntropyLoss |
| hidden_size | 64 |
| num_layers | 1 |
| dropout | 0.3 |
| conv_channels | 64 |
| validation_ratio | 0.2 |
| random_seed | 42 |

CNN-GRU 구조는 크게 아래 순서다.

```text
입력: (batch, 32, 225)
-> Conv1d feature extractor
-> BatchNorm + GELU + Dropout
-> GRU
-> LayerNorm + Dropout
-> Linear classifier
-> 8개 단어 logits
```

학습 중 내부 validation accuracy가 가장 높았던 epoch의 state_dict를 저장한다.

```text
best_epoch: 34
내부 validation 최고 accuracy: 100.00%
외부 validation MVP FULL accuracy: 79.33%
```

내부 validation은 학습 데이터에서 나눈 split이므로 높게 나올 수 있다. 모델 개선 판단은 반드시 외부 validation 결과를 같이 봐야 한다.

## 현재 성능 해석 포인트

현재 결과는 “모델이 완성됐다”가 아니라, “MediaPipe 좌표 기반 baseline이 돌아가고 있고 외부 validation에서 약점이 보인다”로 해석해야 한다.

| 관찰 | 의미 |
|---|---|
| 내부 validation 100% | 학습 데이터 내부 split에서는 구분 가능 |
| 외부 validation 79.33% | 처음 보는 사람/영상에서는 성능 하락 |
| `괜찮다` recall 0.60 | 실제 `괜찮다` 중 40%를 다른 단어로 예측 |
| `감사` recall 0.50 | 실제 `감사` 중 절반만 맞힘 |
| `병원` precision 0.5556 | 모델이 `병원`이라고 말한 것 중 오답 비율이 큼 |
| `자다` f1 0.9831 | 현재 8개 중 가장 안정적인 단어 |

이런 차이는 데이터 수, 사람 차이, 각도 차이, landmark 품질, 동작 유사도 중 하나 또는 복합 원인일 수 있다.

## 중요한 관점

사람은 수어를 볼 때 손 모양, 손의 위치, 손바닥 방향, 손목 회전, 움직이는 방향, 동작 속도, 표정과 맥락을 함께 본다.

현재 모델은 아래처럼 숫자 시퀀스를 본다.

```text
프레임 1: 숫자 225개
프레임 2: 숫자 225개
...
프레임 32: 숫자 225개
```

따라서 모델이 스스로 다음 의미를 학습해야 한다.

```text
손가락이 얼마나 굽혀졌는가
손바닥이 어느 방향을 보는가
손이 몸의 어느 위치에서 움직이는가
손목이 회전하는가
움직임이 빠른가 느린가
```

## `괜찮다` confidence가 낮은 이유 후보

`괜찮다`는 validation 샘플에서 정답으로 맞히더라도 다른 단어보다 confidence가 낮게 나오는 경향이 있다.

가능한 이유는 다음과 같다.

1. `괜찮다` 동작이 손 모양이나 손바닥 방향 차이에 민감할 수 있다.
2. 현재 좌표만으로는 손목 회전이나 손바닥 방향이 약하게 표현될 수 있다.
3. validation 사람인 REAL17/REAL18의 동작 스타일이 train 사람들과 다를 수 있다.
4. 특정 각도, 특히 R/U에서 손가락 landmark가 흔들리거나 가려졌을 수 있다.
5. `괜찮다` train 샘플 수나 사람/각도 다양성이 부족할 수 있다.
6. 비슷한 위치에서 수행되는 다른 단어와 feature가 겹칠 수 있다.

## 팀원 분석 체크리스트

`괜찮다` 또는 confidence가 낮은 단어를 볼 때는 아래를 같이 확인한다.

```text
1. 손가락 landmark가 잘 잡히는가?
2. 손이 몸/얼굴/가슴 기준 어디에 있는가?
3. 손바닥 방향이나 손목 회전이 중요한가?
4. 정면/측면/위쪽 각도에서 같은 동작으로 보이는가?
5. 사람마다 동작 속도와 시작 위치가 다른가?
6. 다른 단어와 헷갈릴 만한 공통 움직임이 있는가?
7. RGB로 보면 쉬운데 좌표만 보면 어려운 동작인가?
```

## 한 줄 정리

현재 모델은 영상을 직접 보는 것이 아니라, 상체와 양손의 3D 좌표 흐름을 보고 있다. 손가락 좌표는 포함되어 있지만, 사람이 직관적으로 보는 손모양/방향/속도/표정 정보는 일부만 간접적으로 반영된다.

## 개선별 전처리/재학습 필요 여부

현재 NPZ는 이미 아래 형태로 만들어져 있다.

```text
X = 32프레임 x 225 feature
225 feature = pose33 + left_hand21 + right_hand21 의 x, y, z 좌표
```

새 feature를 추가하면 입력 차원이 바뀐다.

```text
기존: 32 x 225
변경: 32 x (225 + 추가 feature 수)
```

따라서 feature를 실제 모델 입력에 넣는 실험은 보통 아래 흐름이 필요하다.

```text
새 전처리 -> 새 NPZ 생성 -> 새 모델 학습 -> 새 validation 평가
```

| 모델 분석/개선 내용 | 전처리 다시 필요? | 모델 재학습 필요? | 현재 상태/예시 |
|---|---:|---:|---|
| 현재 NPZ로 전체 accuracy 확인 | X | X | 완료. CNN-GRU 기준 validation 150개 중 119개 정답, accuracy 79.33% |
| 현재 NPZ로 단어별 precision/recall/f1 확인 | X | X | 완료. `cnn_gru_validation_full_metrics.json`에 저장 |
| 현재 NPZ로 파일별 예측/confidence 확인 | X | X | 완료. `cnn_gru_validation_full_predictions.csv`에 저장 |
| 현재 NPZ로 confusion matrix 분석 | X | X | 추가 가능. 예: `괜찮다`가 틀린 4개는 어느 단어로 예측됐는지 확인 |
| confidence 낮은 샘플 심화 분석 | X | X | 추가 가능. 예: `괜찮다` 중 confidence 0.42대 샘플의 공통 특징 확인 |
| 각도별/REAL별 성능 분석 | X | X | 추가 가능. manifest와 prediction CSV 연결. 예: REAL17/REAL18, D/F/L/R/U 각도별 정확도 비교 |
| 손가락 관절 각도 추가 | O | O | 손 모양 구분 강화 목적 |
| 손끝 사이 거리 추가 | O | O | 손가락 벌림/접힘 표현 강화 |
| 손목 기준 상대좌표 추가 | O | O | 사람 위치 차이 완화 |
| 양손 사이 거리 추가 | O | O | 양손 상호작용 표현 강화 |
| 손과 어깨/가슴/얼굴의 상대 위치 추가 | O | O | 몸 기준 위치 정보 강화 |
| 프레임 간 속도/가속도 feature 추가 | O | O | 동작 방향과 빠르기 표현 강화 |
| face/입모양 feature 추가 | O | O | MediaPipe face 추출까지 필요 |
| 좌우 반전 augmentation | △ | O | 원본 재추출 없이 NPZ 단계에서 만들 수도 있음 |
| sequence length 변경 | △ | O | 기존 keypoint가 있으면 재샘플링 가능, 원본 재추출은 아닐 수 있음 |
| 모델 구조만 변경 | X | O | 같은 NPZ로 CNN-GRU/LSTM 등 비교 가능 |

## 전처리 개선 담당자가 확인할 것

전처리를 바꾸는 실험은 모델 입력 데이터 자체를 바꾸는 작업이다. 아래 항목 중 하나라도 바뀌면 기존 checkpoint와 직접 호환되지 않을 수 있다.

| 확인 항목 | 현재 값 | 바꿀 때 주의점 |
|---|---|---|
| landmark layout | `mediapipe_xyz` | pose/hand 순서가 바뀌면 기존 모델 입력 순서와 불일치 |
| feature count | 225 | feature를 추가하면 모델 input_size도 같이 변경 필요 |
| sequence length | 32 | 길이를 바꾸면 모델/웹 window size도 함께 확인 필요 |
| normalize mode | `mediapipe_xyz_body` | 정규화가 바뀌면 기존 학습 분포와 달라짐 |
| missing landmark 처리 | 0 padding | 다른 보간 방식으로 바꾸면 학습 데이터 전체 재생성 필요 |
| frame sampling | 긴 sequence는 32개로 균등 샘플링 | 속도/가속도 feature를 넣을 때 sampling 방식 영향 큼 |
| 수어 구간 | manifest의 `start/end` 사용 | 구간이 틀리면 모델이 준비 동작/마무리 동작까지 학습할 수 있음 |
| MediaPipe confidence | detection/tracking 0.3 | 값을 올리면 품질은 좋아질 수 있지만 누락 프레임 증가 가능 |

전처리 개선 실험을 할 때는 한 번에 여러 항목을 바꾸지 않는 것이 좋다. 예를 들어 `손가락 각도 + 속도 + face`를 동시에 넣으면 성능이 오르거나 내려도 원인을 분리하기 어렵다.

권장 실험 단위:

```text
실험 A: 기존 225 feature + 손가락 관절 각도
실험 B: 기존 225 feature + 손끝 거리
실험 C: 기존 225 feature + 속도 feature
실험 D: 기존 225 feature + 좌우 반전 augmentation
```

각 실험은 반드시 기존 CNN-GRU baseline과 같은 validation set으로 비교한다.

## 학습 개선 담당자가 확인할 것

학습만 바꾸는 실험은 같은 NPZ를 유지하고 모델 구조나 하이퍼파라미터만 바꾸는 작업이다. 이 경우 전처리 재생성은 필요 없지만 모델 재학습은 필요하다.

| 학습 실험 후보 | 전처리 변경 | 재학습 | 확인할 지표 |
|---|---:|---:|---|
| CNN-GRU hidden_size 변경 | X | O | 외부 validation accuracy, 단어별 recall |
| dropout 변경 | X | O | 내부/외부 성능 차이, 과적합 여부 |
| batch_size 변경 | X | O | 학습 안정성, best epoch |
| learning_rate 변경 | X | O | loss 수렴, validation 변동성 |
| LSTM/GRU/CNN-GRU 비교 | X | O | 외부 validation 성능 |
| BiGRU attention/Transformer 실험 | X | O | 데이터 수 대비 과적합 여부 |
| class weight 적용 | X | O | 샘플 수가 적은 단어 recall 개선 여부 |
| augmentation된 NPZ 사용 | △ | O | augmentation 전/후 외부 validation 비교 |

현재 라벨 분포는 균등하지 않다. `가다` 128개, `자다` 100개에 비해 `감사`, `괜찮다`, `병원`, `아프다`는 각 32개다. 따라서 단순 전체 accuracy만 보면 많은 샘플을 가진 단어 쪽 성능이 결과를 끌고 갈 수 있다.

모델 개선 평가는 최소한 아래를 같이 봐야 한다.

```text
1. 전체 accuracy
2. 단어별 precision/recall/f1
3. 특히 괜찮다/감사/아프다/우유 recall
4. confusion matrix
5. confidence가 낮은 정답 샘플
6. confidence가 높은 오답 샘플
7. REAL17/REAL18, D/F/L/R/U 각도별 성능
```

## 고정해야 비교가 가능한 기준

실험 비교를 위해 아래 기준은 가능한 한 고정한다.

| 기준 | 고정 이유 |
|---|---|
| validation dataset | 모델 간 일반화 성능을 같은 조건으로 비교하기 위해 |
| label order | 출력 index 해석 오류 방지 |
| 평가 스크립트 | accuracy/confidence 계산 기준 통일 |
| random seed | train/validation split과 초기값 변동 최소화 |
| baseline 모델 | 개선 효과를 비교할 기준점 필요 |

반대로 아래는 실험 목적에 따라 바꿀 수 있다.

| 변경 가능 항목 | 변경 목적 |
|---|---|
| feature 추가 | 현재 좌표만으로 약한 동작 보강 |
| augmentation | 사람/각도/좌우 차이 대응 |
| model_type | 시계열 패턴 학습 방식 비교 |
| class weight/sampling | 라벨 불균형 완화 |
| sequence length | 동작 길이가 긴 단어 표현력 개선 |

## 개선 실험 기록 방식

실험을 반복할 때는 결과 파일 이름만 남기면 나중에 비교가 어렵다. 최소한 아래 정보를 함께 기록한다.

| 기록 항목 | 예시 |
|---|---|
| 실험명 | `exp_angle_feature_v1` |
| 전처리 변경 | `기존 225 + 손가락 관절 각도 30개 추가` |
| 입력 shape | `(samples, 32, 255)` |
| 모델 구조 | `cnn_gru`, hidden 64, conv 64 |
| 학습 설정 | epoch 50, batch 16, lr 0.001 |
| 내부 validation | best epoch, best accuracy |
| 외부 validation | accuracy, macro f1 |
| 약점 단어 | 예: `괜찮다 recall 0.60 -> 0.80` |
| 실패/주의 | 예: `병원 precision 하락` |

좋은 개선은 전체 accuracy만 올리는 것이 아니라, 약한 단어의 recall/confidence를 올리면서 다른 단어를 크게 망가뜨리지 않는 것이다.

## 다음 2차 분석 예시

1차 분석은 완료되었으므로, 다음 단계는 결과 해석을 더 깊게 하는 것이다.

```text
1. 괜찮다가 틀린 4개 샘플이 어떤 단어로 예측됐는지 확인
2. 괜찮다 confidence 0.42대 샘플에서 landmark 품질이나 각도 문제 확인
3. REAL17과 REAL18 사이 성능 차이 확인
4. D/F/L/R/U 각도별 confidence 평균 비교
5. 사람은 쉽게 맞히는데 모델 confidence가 낮은 샘플 분리
6. 모델도 사람도 어려워하는 샘플 분리
```

## 현실적인 진행 순서

바로 전처리부터 다시 하면 작업 범위가 커진다. 먼저 현재 NPZ와 prediction CSV로 원인을 좁히는 것이 좋다.

```text
1. 현재 결과로 괜찮다 실패/저 confidence 원인 분석
2. 문제가 각도/사람/라벨/landmark 품질 중 무엇인지 확인
3. 필요한 feature만 정해서 추가
4. 새 전처리로 NPZ 재생성
5. 새 모델 학습
6. 기존 모델과 validation accuracy/confidence 비교
```
