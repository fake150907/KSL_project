# 🌺피어나 — 양방향 수어 통역기

> **Pieo:Na — 농인의 목소리가 민원 현장에서 다시 피어나도록**
> ESTsoft 2팀 Team Project · 민원 상담형 키오스크
> 수어 인식 · STT · 상담 요약 · 이메일/카카오톡 알림

**「피어나」** 는 *“말이 닿지 않던 자리에서 농인의 표현이 다시 피어난다”* 는 뜻을 담은 프로젝트 이름입니다.
농인 민원인와 민원 상담원이 한 화면에서 **수어 ↔ 텍스트** 양방향으로 소통할 수 있도록 설계된 민원센터 환경 키오스크 데모입니다. 농인이 수어로 증상·요청을 전달하면 자연어 문장으로 변환해 상담원에게 보여주고, 상담원의 음성은 STT를 통해 텍스트로 농인에게 전달됩니다. 상담이 끝나면 핵심 내용을 이메일/카카오톡으로 요약 발송합니다.

---

## 1. 프로젝트 개요

### 주제 선정 배경

- **국내 청각장애인 약 44만 명** (전체 등록장애인 263만 명 중 16.8%, 2024년 기준)
- **2024년 신규 등록장애 1위(31.7%)가 청각**
- **2026.1 베리어프리 키오스크 단계적 의무화 시행** (장애인차별금지법 시행령)

그러나 민원 현장에는 다음과 같은 문제가 여전히 남아 있습니다.

1. **민원기관 의사소통 장벽** — 민원센터 방문 시 수어 통역사 즉시 배치는 사실상 어려워 필담·구화에 의존
2. **현행 배리어프리 키오스크의 한계** — 음성·점자·높이조절 중심이며, 수어를 입력받는 기능은 거의 부재
3. **법은 시행, 현장 체감은 미흡** — 의무화 이후에도 실제 민원 현장에는 일반 키오스크가 더 많음

> 2020 장애인 실태조사상 청각장애인 의사소통 방법 1위는 ‘말(84.2%)’이지만, 중증 농인에게 **수어는 1차 언어**입니다. 「피어나」는 그 1차 언어를 민원 현장으로 끌어옵니다.

### 왜 「피어나」가 필요한가

| | |
|---|---|
| **민원 의사소통의 중요성** | 접수, 증상 설명, 검사·치료 안내 등 정확한 의사소통이 필수인 공간 |
| **농인 민원인의 불편** | 수어 통역 인력 부족, 음성 중심 안내, 복잡한 민원 용어 |
| **기존 키오스크의 한계** | 접수·수납 중심 기능에 머물러 양방향 상담 기능 부재 |
| **「피어나」의 필요성** | 농인이 수어로 증상·요청을 전달하고, 상담원의 안내를 텍스트로 확인할 수 있는 민원 키오스크 |

### 프로젝트 범위

- **학습 데이터** : WORD ID 3,000개 라벨 (중복 제거 1,723개)
- **적용 대상** : 민원 키오스크 (대학민원센터 환경 가정)
- **발표 형태** : 웹 데모 + 이메일/카톡 알림
- **개발 범위(이번 단계)** : 수어 인식 모델 · STT 음성→텍스트 · Web UI 데모 · 이메일/카톡 알림

### 양방향 통역 흐름

```text
1. 민원 요청        농인이 상담원 연결 버튼으로 상담 요청
2. 수어 → 텍스트    카메라로 수어 인식 후 자연어 문장으로 변환
3. 음성 → 텍스트    상담원의 음성 안내를 STT로 화면에 표시
4. 요약 알림 전송   상담 종료 후 핵심 내용을 이메일/카톡 나에게 발송
```

핵심 시나리오 예시 (3,000 WORD ID 기반):
- 접수 — 「저 + 민원인」
- 증상 — 「저 + 머리 + 아프다」
- 검사 — 「저 + 검사 + 받다 + 원하다」
- 카테고리 커버리지 : 접수 → 증상 → 검사·치료 → 약 → 입퇴원 → 마무리

## 5. 수행 절차 (5단계 파이프라인)

```text
01 데이터 수집·라벨링   → MVP 8개 라벨에서 점진적 확장
02 MediaPipe 전처리    → 손·자세 랜드마크 + 손가락 좌표 확장
03 모델 학습           → 단어 라벨 분류 → 자연어 문장 변환
04 STT 음성 인식       → 상담원 음성 → 실시간 텍스트
05 Web 데모 + 알림     → 양방향 통역 UI + Email 요약 발송
```

핵심 강조 영역 : **MediaPipe 전처리 + STT 음성 인식**이 「피어나」의 두 축이며, 양방향 통역의 정확도와 응답 속도를 좌우합니다.

### 5-1. 데이터 수집 & 라벨링

- **출처** : AIHub 수어 데이터 `Training > REAL > WORD`
- **원천 입력** : REAL_WORD mp4 영상
- **처리 규모** : 1인당 약 100GB 분할 다운로드 & 전처리
- **평균 소요** : 전처리 약 10시간 (PC 환경별 편차 존재)

```text
AIHub REAL WORD → 분할 처리 → MediaPipe 추출 → 시퀀스화(정규화·NPZ) → 검증 & 학습
```

> 대용량 mp4를 그대로 학습하지 않고, 프레임별 좌표 시퀀스로 변환해 조명·배경 변화에 덜 민감한 입력 데이터로 구성합니다.

### 5-2. 전처리 개선 과정

| 구분 | 초기 MVP | 개선 버전 |
|---|---|---|
| 라벨 수 | **8개** | **1,723개** (WORD ID 3,000개 중복 제거) |
| 시나리오 커버리지 | 민원 대화 커버 불가 | 접수~입퇴원 시나리오 커버 |
| Face Landmarks | 시도했으나 불안정 | 제외 (안정적 추출 어려움) |
| Hand 좌표 | 기본 21개 → 변별력 부족 | 관절별 좌표 추가 보강 |

### 5-3. MediaPipe 기반 수어 전처리

```text
① 프레임 캡처     OpenCV로 카메라 프레임 획득 (BGR → RGB)
② Hand·Pose 추출  손/상반신 자세 키포인트만 검출 (Face 제외)
③ 좌표 정규화·확장 어깨 너비 기준 스케일링 + 손가락 좌표 보강
④ 시퀀스 구성     타임스텝별 좌표 누적 → 수어 단어 단위 입력 벡터
```

| 키포인트 | 사용 여부 |
|---|---|
| Hand Landmarks (양손 × 21 + 추가 좌표) | ✅ 사용 — 관절별 보강으로 변별력 ↑ |
| Pose Landmarks (상반신, 어깨·팔꿈치·손목) | ✅ 사용 |
| Face Landmarks | ❌ 미사용 — 안정적 추출 어려움 |

> 픽셀이 아닌 **좌표**를 학습하므로 조명·의상·배경 변화에 강하고 학습 효율이 높습니다.

### 5-4. STT 음성 인식 흐름

```text
🎤 음성 입력            상담원 마이크로부터 연속 오디오 스트림 수집
📡 실시간 스트리밍      짧은 청크 단위로 STT 엔진 전송, latency 최소화
📝 텍스트 변환·보정     민원 용어 사전 보강, 종결 어미 정제
🖥️ 화면 출력           농인 민원인 화면 즉시 표시 + 종료 시 요약 발송
```

엔진 : **Web Speech API** 기반 실시간 변환

## 6. 수행 결과

### 6-1. 시연 시나리오

| 단계 | 입력 수어 | 처리 |
|---|---|---|
| 01 접수 | 「저 + 민원인」 | 접수 의도 분류 |
| 02 증상 설명 | 「저 + 머리 + 아프다」 | 자연어 문장 생성 |
| 03 검사 요청 | 「저 + 검사 + 받다 + 원하다」 | 상담원 STT 응답 |

### 6-2. 수어 인식 모델 학습 결과

MediaPipe 좌표 시퀀스 기반 **CNN-GRU** 분류 모델로 학습한 「피어나」의 핵심 성능 지표입니다.

| 항목 | 값 |
|---|---|
| **학습 모델 / Model** | **CNN-GRU** |
| **정확도 / Test Accuracy** | **95.68 %** |
| **최종 Validation Loss** | **1.5562** |
| **학습 단어 수 / Classes** | **3,000** |
| **총 학습 Epoch** | **300** |

#### 학습 곡선 (Loss & Accuracy)

![CNN-GRU Loss & Accuracy Curve](assets/slide15_그림_28.png)

- **Loss Curve (좌)** — Epoch 200 → 300 구간에서 Train Loss가 1.97 → 1.77, Val Loss가 1.68 → 1.56 까지 안정적으로 하강합니다. 260 epoch 부근에서 한 번 큰 폭의 손실 감소가 일어나는 학습 분기점이 관찰됩니다.
- **Accuracy Curve (우)** — Train Acc가 88% → 94% 대로 상승하는 동안 Val Acc는 92% → 95.68%까지 도달하며, **Val Acc ≥ Train Acc** 양상으로 과적합 없이 일반화가 안정적으로 유지되었습니다.

## 7. 자체평가

### 성과 (Achievements)

- **CNN-GRU 모델로 Test Accuracy 95.68% 달성** — 3,000 클래스 대규모 분류에서 상용 가능한 성능 확보
- **Val Loss 1.5562까지 안정 수렴** — 300 epoch에 걸쳐 Train/Val 곡선의 격차가 좁혀지며 일반화 검증
- **MediaPipe 좌표 기반 일반화** — 픽셀 학습 대비 카메라·배경에 강건한 입력 설계
- **STT 양방향 흐름 구현** — 농인 호출부터 이메일 요약 발송까지 end-to-end 검증
- **민원 시나리오 단어 커버리지** — MVP 8개 → **1,723개 라벨**로 확장
- **팀 환경 표준화** — 공통 라이브러리 버전·분기 전략으로 전처리/추론 gap 최소화

### 한계점 (Limitations)

- **데이터 다양성 부족** — 특정 화자·각도 위주 데이터로 편향 가능성
- **실시간 처리 지연** — 카메라 → 추출 → 추론 파이프라인 latency 추가 최적화 필요
- **문맥 단위 통역 한계** — 단어 단위 매핑 위주, 문장 어순·조사 자연스러움 개선 필요
- **STT 민원 용어 정확도** — 전문 용어·약어 오인식 시 사후 보정 사전 보강 필요

### 향후 개선 방향

| 영역 | 방향 |
|---|---|
| 📈 데이터 다양화 | 연령·성별 균형, 조명·각도 변동, 수어 사투리 반영 |
| ⚡ 추론 경량화 | TFLite·ONNX 변환, 프레임 샘플링, GPU/엣지 분기 |
| 📚 문장 단위 통역 | 소형 LLM 후처리, 조사·어미 보정, 민원 의도 분류 |
| 🔔 카카오톡 알림 도입 | 플러스친구 정식 알림, 자동 푸시, 발송 이력 관리 |
